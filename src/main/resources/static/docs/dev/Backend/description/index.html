Backend - Introducción

Descripción General

El backend de CLIAS está construido en Spring Boot + Kotlin y sigue una arquitectura en capas (Controller → Service → Repository → Entity/DTO) con Spring Security (JWT), Spring Data JPA y Thymeleaf para páginas informativas mínimas. Soporta tareas programadas y auditoría JPA.

Punto de entrada

  • RegistroApplication.kt con @SpringBootApplication, @EnableJpaAuditing y @EnableScheduling.

Funcionalidades Actuales

  • Autenticación y cuentas: registro/login JWT, gestión de perfiles (AuthController, CuentaUsuarioController).
  • Pacientes y profesionales: CRUD y flujo clínico básico (PacienteController, MedicoController).
  • Examen VPH: registro, seguimiento y resultados (ExamenVphController, EvolucionController).
  • Encuestas y datos complementarios: SUS e información socioeconómica (EncuestaSusController, InformacionSocioeconomicaController).
  • Dispositivos y estados (SISA): vinculación app-usuario, QR, estados y dispositivos de automuestreo registrados (DispositivoAppUsuarioController, DispositivoRegistradoController, EstadoDispositivoController, CodigoQRController).
  • Ubicación/Mapa: captura/consulta de ubicaciones de uso de la app (UbicacionController).
  • Notificaciones: envío PUSH vía Firebase y programación de notificaciones (NotificacionController, PushNotificationService, NotificacionService).
  • Recursos y archivos: carga/descarga y contenidos de apoyo (ArchivoController, RecursoController).
  • Salud sexual: contenidos y endpoints temáticos (SaludSexualController).
  • Sesión de chat (soporte/asistencia): almacenamiento/consulta de sesiones (SesionChatController).
  • Páginas web: contenido informativo bajo Thymeleaf (WebController, templates/*).

Estructura del proyecto

Árbol de paquetes (vista lógica)​

  • com.crisordonez.registro
    • configuration
      • SecurityConfig → reglas de seguridad, endpoints públicos/privados.
      • JwtAuthenticationFilter → validación de token y carga de usuario.
      • CorsConfig → orígenes permitidos para web y app móvil.
      • FirebaseConfig → inicialización de SDK para notificaciones PUSH.
      • AppExceptionHandler → manejo unificado de errores (HTTP -> JSON).
    • controller
      • AuthController, CuentaUsuarioController, AdministradorController
      • PacienteController, MedicoController
      • ExamenVphController, EvolucionController
      • EncuestaSusController, InformacionSocioeconomicaController
      • DispositivoAppUsuarioController, DispositivoRegistradoController, EstadoDispositivoController
      • CodigoQRController, ArchivoController, RecursoController
      • NotificacionController, SesionChatController, UbicacionController
      • PagesController/WebController (páginas Thymeleaf)
    • service
      • Servicios por dominio + sus *ServiceInterface
      • Notificaciones: PushNotificationService, NotificacionService
      • Programación: schedule/* para tareas con @Scheduled
    • repository
      • Repositorios Spring Data: PacienteRepository, MedicoRepository, ExamenVphRepository, EvolucionRepository, DispositivoAppUsuarioRepository, DispositivoRegistradoRepository, EstadoDispositivoRepository, UbicacionRepository, NotificacionRepository, NotificacionProgramadaRepository, SesionChatRepository, RecursoRepository, ArchivoRepository, EncuestaSusRepository, InformacionSocioeconomicaRepository, CuentaUsuarioRepository, AdministradorRepository, etc.
    • model
      • entities/ → entidades JPA (tablas y relaciones).
      • dtos/ → DTOs de entrada/salida.
      • requests/ y responses/ → contratos de API.
      • enums/ → estados y catálogos (p. ej., estados de dispositivo).
      • mapper/ → conversión Entity ↔ DTO (cuando aplica).
      • errors/ → tipos de error específicos de dominio.
    • utils
      • JwtUtil (firma/validación), MensajesNotificacion (plantillas), utilidades comunes.
    • RegistroApplication.kt

Recursos​

  • src/main/resources
    • templates/ (Thymeleaf): index.html, proyecto.html, quienes-somos.html, preguntas-frecuentes.html, contactos.html, _fragments/*.
    • static/ (CSS/imagenes/web), favicon.ico
    • application.properties / application.yml
    • logback.xml
    • Credenciales Firebase (JSON) para FCM en entorno seguro.

Diagrama (Alto nivel)​

Regla de diseño: Por cada entidad/tabla principal del dominio hay al menos:

  • Un Controller (expone endpoints REST del módulo),
  • Un Service (lógica de negocio del módulo),
  • Un Repository (acceso a datos),
  • Y su Entity/DTOs correspondientes.
    Los endpoints se agrupan por módulo según la sección “Funcionalidades actuales” más arriba.
    Excepciones: algunos controladores agregan varias entidades (p.ej., Notificaciones usa Notificacion y NotificacionProgramada), y WebController sirve plantillas Thymeleaf (no mapea 1:1 a una sola tabla).
