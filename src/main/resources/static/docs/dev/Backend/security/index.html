Backend - Seguridad

Enfoque general​

  • Autenticación: el usuario se loguea (p. ej. /api/auth/login) y recibe un JWT.
  • Autorización: en cada request posterior el cliente envía el header
    Authorization: Bearer <token>.
    Un filtro (JwtAuthenticationFilter) valida el token y, si es correcto, autentica la petición en el SecurityContext.
  • Sin sesión de servidor: el token contiene la identidad; el servidor no guarda estado de sesión.

Flujo (alto nivel)​

Componentes clave de la configuración​

SecurityConfig​

Puntos esenciales de la configuración:

  • CORS: habilitado con CorsConfigurationSource (actualmente allowedOriginPatterns=["*"] ).
  • CSRF: deshabilitado (csrf.disable()), típico en APIs JWT.
  • Autorización: reglas con authorizeHttpRequests { … } donde se define qué rutas son públicas (permitAll) o requieren rol (hasRole).
  • Filtro JWT: addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class) para validar el token en cada request.
  • Autenticación:
    • UserDetailsService propio (CuentaUsuarioDetailService)
    • DaoAuthenticationProvider + BCrypt para comparar contraseñas hasheadas
    • AuthenticationManager con ese provider

JwtAuthenticationFilter​

  • Extrae el token del header Authorization.
  • Lo valida (firma, expiración, etc.) y, si es válido, construye un UsernamePasswordAuthenticationToken con el usuario/roles y lo deja en el SecurityContext.
  • Si no hay token válido → la request llegará no autenticada y se aplican las reglas.

CuentaUsuarioDetailService​

  • Carga el usuario desde la BD (por email/username).
  • Debe devolver un UserDetails con authorities.
  • Si usas hasRole("ADMIN"), la autoridad esperada es ROLE_ADMIN.

Cómo proteger / agregar endpoints (paso a paso)​

  1. Crea el endpoint en tu @RestController o @Controller.

  2. Decide si será:

    • público → permitAll()
    • autenticado → sin regla específica; caerá en anyRequest().authenticated()
    • por rol → hasRole("ADMIN") / hasRole("USER"), etc.

  3. Agrega la regla en authorizeHttpRequests (cuanto antes, más prioridad):

    // Todo /api/reportes/* solo ADMIN
    registry.requestMatchers("/api/reportes/**").hasRole("ADMIN")

    // Solo lectura pública en /api/medicos; resto autenticado
    registry.requestMatchers(HttpMethod.GET, "/api/medicos/**").permitAll()

  4. Verifica que tus usuarios tengan la autoridad ROLE_XXX compatible.

  5. Prueba con y sin token.

Orden importa: Spring evalúa matchers en el orden declarado. Pon las rutas más específicas arriba.
