Backend - Despliegue

En este apartado se explica cómo levantar el backend de CLIAS en local a partir de la clonación del repositorio en github.

Requisitos​

  • JDK 21 (Amazon Corretto o Temurin)
  • Maven 3.9+
  • PostgreSQL 14+
  • Git 2.40+
  • Docker / Docker Compose
  • IDE recomendado: IntelliJ IDEA (Community o Ultimate)

Verifica versiones:

java -version
mvn -v
psql --version
docker --version
docker compose version

Despliegue Local​

Clonar el proyecto​

Con el siguiente comando copiamos el repositorio:

git clone https://github.com/chr1s23/TelemedicinaBE.git
cd TelemedicinaBE

Recomendamos abrir la carpeta del proyecto en IntelliJ IDEA, ya que este IDE resuelve las dependencias de Maven rápidamente.

Después de tener el repositorio clonado, es necesario que crear la base de datos en PostgreSQL para un despliegue local.

PostgreSQL local​

Asegúrate de tener PostgreSQL corriendo en localhost:5432. Puedes trabajar directamente con la BD postgres y el usuario postgres. Si necesitas crear un usuario/BD específicos, ejecuta algo como:

-- Opcional: solo si quieres una BD propia (no obligatorio)
CREATE DATABASE clias;
CREATE USER clias WITH ENCRYPTED PASSWORD 'clias';
GRANT ALL PRIVILEGES ON DATABASE clias TO clias;

Configurar acceso a BD para local​

En el repo, el application.properties ya trae dos variantes. Para local, usa las propiedades directas (descomentando estas líneas y ajustando si hace falta):

# (LOCAL) USAR ACCESO DIRECTO A POSTGRES LOCAL
spring.datasource.url=jdbc:postgresql://localhost:5432/postgres
spring.datasource.username=postgres
spring.datasource.password=postgres

# (PRODUCCIÓN) ESTA VARIANTE QUEDA DESACTIVADA EN LOCAL
# spring.datasource.url=${SPRING_DATASOURCE_URL}
# spring.datasource.username=${SPRING_DATASOURCE_USERNAME}
# spring.datasource.password=${SPRING_DATASOURCE_PASSWORD}

nota

Deja el resto de parámetros tal como están en el repo (HikariCP, JPA, multipart, etc.).
No necesitas tocar driver-class-name, ddl-auto, ni los timeouts.

Ejecuta la app de forma local​

Con IntelliJ IDEA

  1. Abre la clase principal anotada con @SpringBootApplication.

  2. Ejecuta Run ▶️ (o Shift+F10)

Despliegue en Producción con Docker Compose​

Este despliegue levanta PostgreSQL y el Backend en contenedores. El backend se conecta a la BD usando el hostname db dentro de la red de Docker.

Archivo docker-compose.yml​

Usa exactamente este contenido:

version: "3.9"

services:
  db:
    image: postgres
    container_name: postgres_db
    restart: always
    environment:
      POSTGRES_DB: postgres
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  app_be:
    build: .
    container_name: telemedicina_be
    restart: always
    depends_on:
      - db
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/postgres
      SPRING_DATASOURCE_USERNAME: postgres
      SPRING_DATASOURCE_PASSWORD: postgres
      TZ: America/Guayaquil
    ports:
      - "8080:8080"
    volumes:
      - /etc/localtime:/etc/localtime:ro

volumes:
  postgres_data:
    external: true
    name: telemedicinabe_postgres_data

Crear el volumen externo (requerido)​

El docker-compose usa un volumen externo llamado telemedicinabe_postgres_data. Si no existe, créalo una sola vez:

docker volume create telemedicinabe_postgres_data

Construir e iniciar​

Ejecuta desde la carpeta donde está el docker-compose.yml (y el Dockerfile del backend):

docker compose up -d --build

  • La BD quedará disponible en el host en localhost:5432.

  • La API quedará publicada en http://localhost:8080.

Configuración de la app en producción​

No es necesario editar application.properties para producción: el backend toma las credenciales de las variables de entorno definidas en app_be.environment:

  • SPRING_DATASOURCE_URL=jdbc:postgresql://db:5432/postgres

  • SPRING_DATASOURCE_USERNAME=postgres

  • SPRING_DATASOURCE_PASSWORD=postgres

nota

Si quieres cambiar usuario/clave o el nombre de la BD en producción, ajusta los valores en el servicio db y en app_be.environment.

Logs y salud​

docker compose logs -f app_be
docker compose logs -f db
