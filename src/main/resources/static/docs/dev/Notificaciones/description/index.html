Notificaciones - Introducción

Tipos de notificación activos​

En el backend solo existen automatizaciones para los tres primeros tipos.

  • RESULTADO
    Se envía cuando la doctora sube el resultado en el frontend administrativo (web). Dispara un push al/los dispositivo(s) del usuario y además inicia un plan de notificaciones programadas (recordatorios) relacionados a completar la encuesta SUS / revisar resultado.

  • RECORDATORIO_NO_EXAMEN
    Recordatorio para usuarias que aún no tienen un examen registrado (no aparece un examen asociado al paciente). Se envía de forma programada; si detecta que ya existe examen, se desactiva automáticamente.

  • RECORDATORIO_NO_ENTREGA_DISPOSITIVO
    Recordatorio para confirmar la entrega del dispositivo de automuestreo. Si el backend detecta que el examen ya fue entregado (contenido disponible), se desactiva automáticamente. La app permite al usuario confirmar y solicitar que se deje de recordar.

  • BIENVENIDA (opcional, sin automatizaciones en backend)
    La app móvil puede crear una notificación de bienvenida (vía /notificaciones) y además mostrar un banner local. El backend la persiste como cualquier notificación, pero no tiene lógica programada asociada (no inicia plan recurrente).

Resumen de alcance

  • Backend con lógica de negocio/automatización: RESULTADO, RECORDATORIO_NO_EXAMEN, RECORDATORIO_NO_ENTREGA_DISPOSITIVO.
  • Backend “pasivo” (solo persiste/empush-a): BIENVENIDA (cuando el frontend móvil lo solicita).
  • NUEVO_RECURSO: No se ha usado por lo que está desactivado en el frontend móvil.

Estructura de la notificación​

Entidad/Respuesta base (persistencia y API)​

Una notificación “real” que ve la usuaria se modela y expone con los atributos:

  • publicId (UUID): Identificador público de la notificación.
  • cuentaUsuarioPublicId (UUID): A quién pertenece.
  • titulo (String): Encabezado visible en el push/listado.
  • mensaje (String): Bontenido visible.
  • tipoNotificacion (Enum): RESULTADO, RECORDATORIO_NO_EXAMEN, RECORDATORIO_NO_ENTREGA_DISPOSITIVO, BIENVENIDA.
  • tipoAccion (Enum): Tipo de acción asociada (interno de la app).
  • accion (String): Ruta interna o URL externa a abrir cuando la usuaria toca la notificación.
  • fechaCreacion (DateTime): Momento en que se creó la notificación (o inicio en el caso de plantillas programadas).
  • notificacionLeida (Boolean): Estado de lectura (se marca al tocarla).

Uso en backend al crear una notificación “real”

  • El servicio NotificacionService.crearNotificacion(...) persiste y dispara un push:
    pushNotificacionService.enviarPushFCM(cuentaUsuario.publicId, notificacionResponse)

    Esto envía a todos los dispositivos FCM válidos asociados a la cuenta.

Plantilla programada (scheduler de recordatorios)​

Para notificaciones recurrentes (recordatorios) se maneja una plantilla programada, con atributos clave:

  • tipoNotificacion: el tipo que se va a ir instanciando (p. ej., RESULTADO, RECORDATORIO_NO_EXAMEN, RECORDATORIO_NO_ENTREGA_DISPOSITIVO).
  • titulo, mensaje, tipoAccion, accion: contenido base que heredan las instancias reales.
  • fechaInicio (DateTime): desde cuándo corre el plan.
  • proxFecha (DateTime): próxima ejecución.
  • limiteFecha (DateTime, opcional): fecha máxima para enviar recordatorios.
  • programacionActiva (Boolean): habilita/inhabilita la programación.

Cadencia y autocancelación

  • Cadencia dinámica al procesar:
    • 0–30 días desde fechaInicio: cada 3 días
    • 31–60 días: cada 7 días
    • >60 días: se detiene
    • Si limiteFecha se cumplió, también se detiene.
  • Reglas de apagado automático antes de instanciar:
    • RECORDATORIO_NO_EXAMEN: si ya existe examen del paciente → desactivar.
    • RECORDATORIO_NO_ENTREGA_DISPOSITIVO: si el examen consta entregado → desactivar.
    • RESULTADO: si la usuaria ya respondió encuesta SUS → desactivar.

La ejecución se realiza en NotificacionService.procesarNotificacionesProgramadas(), que:

  1. valida condiciones,
  2. crea la notificación real (y por ende dispara el push),
  3. actualiza proxFecha o apaga la programación.

Registro de dispositivos y envío de push​

Para poder recibir push, la app móvil:

  1. Solicita permisos FCM y obtiene token FCM.
  2. Llama a POST /dispositivo con:
    • usuarioPublicId (UUID)
    • fcmToken (String)
  3. El backend asocia el token al usuario (puede haber múltiples dispositivos).
  4. Al crear una notificación, el backend envía el push a todos los tokens activos del usuario.

En Flutter, esto vive en NotificationService.registrarTokenFCM(...).
En backend, el envío se resuelve vía PushNotificacionServiceInterface.enviarPushFCM(...).
