Notificaciones - Envío manual (Script)

¿Para qué sirve?​

Este script permite enviar notificaciones masivas a las usuarias de forma inmediata (no programada) cuando se requiere comunicar algo urgente u oportuno (por ejemplo, actualización de la app, avisos operativos, mantenimiento, etc.).
Se apoya en la misma API del backend que usa la app para crear notificaciones, y selecciona a las destinatarias consultando la base de datos para obtener los publicId de cuentas con token FCM válido.

Cuándo usarlo

  • Avisos globales no cubiertos por las automatizaciones.
  • Campañas puntuales (ej. actualización de app).
  • Casos donde no se desea crear ni mantener una notificación programada.

Flujo de alto nivel​

  1. Carga variables de entorno (credenciales de BD, URL API).
  2. Consulta la tabla dispositivo_app_usuario y reúne usuarios únicos con fcm_token válido.
  3. Construye el payload de notificación (título, mensaje, tipo, acción).
  4. Envía una solicitud HTTP por usuaria al endpoint /notificaciones.
  5. Hace pausas cortas entre envíos para no saturar al backend/FCM.
  6. Imprime éxitos y errores por cada intento.

Requisitos​

  • Python 3.10+
  • Dependencias: psycopg2, requests, python-dotenv
  • Variables de entorno:
    • SPRING_DATASOURCE_URL (postgresql://user:pass@host:5432/db)
    • SPRING_DATASOURCE_USERNAME
    • SPRING_DATASOURCE_PASSWORD
  • Endpoint del backend accesible (por defecto: https://clias.ucuenca.edu.ec/notificaciones).

Instalación rápida

python -m venv .venv && source .venv/bin/activate
pip install psycopg2-binary requests python-dotenv

Selección de destinatarias​

Se consideran todas las cuentas que posean al menos un dispositivo con fcm_token no vacío:

Esto evita envíos a cuentas sin dispositivos válidos y deduplica por usuaria.

Estructura de la notificación (payload)​

Campos que el script envía al backend (equivalentes a NotificacionRequest):

  • cuentaUsuarioPublicId (UUID en texto) — destinataria
  • tipoNotificacion (String) — tipo lógico de la notificación
  • titulo (String)
  • mensaje (String)
  • tipoAccion (String) — categoriza la acción
  • accion (String) — ruta interna (p. ej. https://miapp.com/...) o URL externa

Rutas internas vs externas

  • Si accion empieza con https://miapp.com/, la app navega dentro de la app (pestañas, pantallas internas).
  • En caso contrario, la app abre un diálogo con botón Navegar para abrir la URL externa en el navegador del dispositivo.

Este diseño aporta escalabilidad: nuevos destinos se agregan sin tocar la lógica base, solo cambiando el valor de accion.

Script de referencia (general)​

Nota: ajusta ENDPOINT, HEADERS y los textos de la notificación según el caso.
Si tu API requiere JWT, agrega Authorization al diccionario HEADERS.

import os
import psycopg2
import requests
from dotenv import load_dotenv
import time

# 1) Variables de entorno
load_dotenv()
DB_URL = os.getenv("SPRING_DATASOURCE_URL")  # postgresql://user:pass@host:5432/dbname
DB_USER = os.getenv("SPRING_DATASOURCE_USERNAME")
DB_PASS = os.getenv("SPRING_DATASOURCE_PASSWORD")

# 2) Endpoint backend
ENDPOINT = "https://<dominio>/notificaciones"
HEADERS = {
    "Content-Type": "application/json",
    # "Authorization": "Bearer <TOKEN_JWT>",  # si se necesita
}

# 3) Contenido de la notificación (ejemplo: actualización de app)
TITULO = "Actualiza tu App SISA"
MENSAJE = (
    "Ya está disponible la nueva versión de la app. Ingresa a https://<dominio>/ "
    "y ve al apartado 'Descarga la Aplicación Móvil'."
)
TIPO = "ACTUALIZACION_APP"
TIPO_ACCION = "NUEVO_RECURSO"
ACCION = "https://clias.ucuenca.edu.ec/"

def obtener_usuarios():
    conn = psycopg2.connect(DB_URL, user=DB_USER, password=DB_PASS)
    cursor = conn.cursor()
    query = '''
        SELECT DISTINCT usuario_public_id::text
        FROM dispositivo_app_usuario
        WHERE fcm_token IS NOT NULL AND fcm_token <> ''
    '''
    cursor.execute(query)
    resultados = cursor.fetchall()
    cursor.close()
    conn.close()
    return [fila[0] for fila in resultados]

def enviar_notificacion(num_user, public_id):
    payload = {
        "cuentaUsuarioPublicId": public_id,
        "tipoNotificacion": TIPO,
        "titulo": TITULO,
        "mensaje": MENSAJE,
        "tipoAccion": TIPO_ACCION,
        "accion": ACCION
    }
    try:
        response = requests.post(ENDPOINT, json=payload, headers=HEADERS, timeout=20)
        if response.status_code in (200, 201):
            print(f"{num_user}) [OK] Notificación enviada a {public_id}")
        else:
            print(f"[X] Error {response.status_code} al enviar a {public_id}: {response.text}")
    except Exception as e:
        print(f"[X] Excepción con {public_id}: {e}")

def main():
    usuarios = obtener_usuarios()
    print(f" Total usuarios a notificar: {len(usuarios)}")

    #  Descomenta para ejecutar los envíos
    # for i, public_id in enumerate(usuarios, start=1):
    #     enviar_notificacion(i, public_id)
    #     time.sleep(0.4)  # Evita saturar el backend/FCM

if __name__ == "__main__":
    main()

Por qué pausa (time.sleep(0.4))
Para limitar el rate y evitar picos que generen errores de red, timeouts o bloqueos temporales en el backend/FCM.

Ejemplo real: actualización de app​

El ejemplo incluido envía una notificación con ACCION = "https://clias.ucuenca.edu.ec/".
En la app, esto se muestra con un diálogo y un botón Navegar que abre el sitio del proyecto para descargar la nueva versión.

Ventaja: permite campañas de actualización sin publicar una nueva versión de la app.

Buenas prácticas​

  • Probar en entorno de staging con una muestra pequeña antes del envío masivo.
  • Mantener listas de exclusión (si corresponde) y revisar duplicados.
  • Autorización: si la API lo requiere, usar JWT con privilegios adecuados y rotarlo al finalizar.
  • Registrar un log de resultados por public_id (éxito/error) para auditoría y reintentos.
  • Evitar mensajes ambiguos; incluir call-to-action claro (ej. “Navegar”, “Actualizar”).
  • En campañas grandes, considerar lotes (batching) y métricas de entrega/lectura.

Variantes útiles (opcionales)​

  • Filtro por cohorte (ej. por fecha de registro o provincia) agregando cláusulas a la consulta SQL.
  • Lista acotada: leer public_id desde un archivo (CSV/JSON) para envíos segmentados.
  • Mensaje templado: cargar TITULO/MENSAJE desde un .env o un JSON de plantillas.
