Chatbot - Integración y Despliegue

Esta página documenta el estado del chatbot en un servidor de producción, conjuntamente con cómo operan sus componentes (Rasa + acciones con LLM phi3-mini-128k-onnx vía onnxruntime-genai) junto al backend y PostgreSQL.

Host (ejemplo): ubuntu
Ruta del proyecto: ~/projects/chatbot_fixed/
Contenedores activos:

  • rasa-core → rasa/rasa:3.6.2-full (puerto 5005)
  • rasa-actions → acciones personalizadas (puerto 5055)
  • telemedicina_be → backend (puerto 8080)
  • postgres_db → BD (puerto 5432)

Estructura del proyecto​

chatbot/
├─ actions/ # Acciones personalizadas (Python)
│ ├─ cpu_and_mobile/ # Modelo phi3-mini-128k-onnx (para onnxruntime-genai)
│ ├─ actions.py # Acciones (fallback/generación con LLM)
│ ├─ Dockerfile # Imagen para 'rasa-actions'
│ └─ requirements.txt
├─ data/ # Datos de entrenamiento NLU/Core
│ ├─ nlu.yml
│ ├─ rules.yml
│ └─ stories.yml
├─ models/ # Modelos Rasa entrenados (.tar.gz)
├─ results/ # Reportes de entrenamiento/pruebas
├─ tests/ # Pruebas NLU / Historias
├─ config.yml # Pipeline NLU + políticas Core
├─ credentials.yml # Canales (REST, etc.)
├─ domain.yml # Intents, entities, slots, responses, actions
├─ endpoints.yml # Endpoints (actions, tracker, etc.)
└─ docker-compose.yml # Orquestación de servicios

Integración real (App ↔ Backend ↔ Rasa)​

  • La app SISA (Flutter) envía los mensajes al backend (Spring Boot) autenticado con JWT.
  • El backend reenvía el mensaje a Rasa usando el canal REST (/webhooks/rest/webhook) y persiste la sesión en PostgreSQL (SesionChat y, si se habilita, ChatMessage).
  • Rasa decide la respuesta:
    • Si hay intent claro, responde con plantillas o acciones.
    • Si no, entra fallback → llama a rasa-actions; allí actions.py usa phi3-mini-128k-onnx para generar una respuesta controlada por prompt (seguridad temática).

Puntos clave para integrar y desplegar​

1) docker-compose.yml — Orquestación​

¿Qué hace?

  • Levanta 2 contenedores:
    • rasa-core (imagen rasa/rasa:3.6.2-full) expuesto en :5005 con --enable-api.
    • rasa-actions (Dockerfile propio en actions/) expuesto en :5055.
  • Monta el directorio del proyecto dentro de /app para que Rasa vea config.yml, domain.yml, data/, models/.
  • Une todo en una red interna (rasa-network).

¿Por qué importa?

  • Así el backend solo necesita hablar con rasa-core:5005 (y Rasa llama internamente a rasa-actions:5055).
  • Separar Core y Actions permite versionar y escalar la lógica de acciones sin tocar el NLU.

¿Qué debes cambiar?

  • Si no quieres exponer Rasa públicamente, elimina/evita el mapeo 5005:5005 y haz que solo el backend acceda a Rasa dentro de la red de Docker.
  • Si mueves el proyecto, ajusta el volumen (- ./:/app).
  • Si cambias de puerto de acciones, actualiza endpoints.yml (ver §4).

2) config.yml — Pipeline NLU y Políticas de diálogo​

¿Qué hace?

  • Define pipeline en español:
    • WhitespaceTokenizer + RegexFeaturizer: tokenización y patrones.
    • CountVectorsFeaturizer (palabras y char n-grams 2–4): robustez ante errores tipográficos y variaciones.
    • DIETClassifier: intents/entidades (200 épocas, LR=0.002).
    • ResponseSelector: retrieval intents (pregunta → respuesta).
    • FallbackClassifier: umbrales threshold=0.3, ambiguity_threshold=0.02.
  • Define políticas:
    • RulePolicy con core_fallback_action_name: action_default_fallback (umbral 0.53).
    • TEDPolicy + Memoization para el flujo.
    • UnexpecTEDIntentPolicy para detectar intenciones inesperadas.

¿Por qué importa?

  • Los umbrales controlan cuándo se considera incierta una predicción → aquí entra el fallback que llama al LLM (seguro con prompt).
  • Los char n-grams hacen que el chatbot entienda variantes y faltas de ortografía.

¿Qué debes ajustar?

  • Si el bot responde con demasiada facilidad, sube FallbackClassifier.threshold (menos “valiente”).
  • Si el bot entra en fallback muy pronto, baja ese valor (más “valiente”).
  • Ajusta epochs/learning_rate si reentrenas y notas sobreajuste o bajo recall.

3) domain.yml — Contrato del asistente​

¿Qué contiene?

  • intents, entities, slots, responses, actions.
  • Lista la acción action_default_fallback y, si aplica, action_consult_info.

¿Por qué importa?

  • Es el contrato entre NLU/Core y tu código de acciones.
    Si no declaras una acción aquí, Rasa no la invocará aunque exista en actions.py.

¿Qué debes cambiar?

  • Cada vez que agregues una acción nueva en Python, declárala en domain.yml.
  • Mantén responses: para mensajes estáticos (más rápidos y controlables que el LLM).

4) endpoints.yml y credentials.yml — Conexiones​

endpoints.yml

  • Apunta acciones a http://rasa-actions:5055/webhook (dentro de la red Docker).
  • Si cambias el puerto/servicio de acciones, actualiza aquí.

credentials.yml

  • Activa el canal rest: para POST /webhooks/rest/webhook.
  • No expongas otros canales (facebook, telegram, etc.) si no los usas.

5) actions/actions.py — Fallback con LLM (ONNX)​

¿Qué hace?

  • Carga phi3-mini-128k-onnx (vía onnxruntime_genai) y su tokenizador.
  • Define un prompt restrictivo (temas permitidos, tono, out-of-domain).
  • Implementa:
    • action_consult_info: generación bajo demanda.
    • action_default_fallback: se dispara en incertidumbre; genera con LLM y revierte el último user_utterance para no “contaminar” la historia.

¿Por qué importa?

  • El LLM no responde cualquier cosa: el prompt obliga a ceñirse a salud sexual/VPH y a recomendar acudir a profesionales cuando corresponda.
  • Usar ONNX permite despliegue portable (sin GPU), a costa de calentamiento inicial.

¿Qué debes cambiar?

  • Ruta del modelo si lo mueves: /app/cpu_and_mobile/phi3-mini-128k-onnx.
  • Parámetros de generación (temperature, max_new_tokens) según calidad/latencia.
  • Refuerza el prompt si detectas respuestas fuera de tema.

6) data/ y models/ — Entrenamiento​

data/

  • nlu.yml, rules.yml, stories.yml — el dataset.
    Cambios aquí requieren reentrenar.

models/

  • Paquetes .tar.gz entrenados (rasa train).
    La versión activa es la que Rasa encuentra en models/ al arrancar.
tip
  • Versiona los .tar.gz estables (nómbrelos con fecha).
  • Guarda matrices de confusión y errores de intent (carpeta results/) tras cada entrenamiento.

7) Seguridad y exposición​

  • En producción, no expongas 5005/5055 al exterior; deja que solo el backend los vea por red interna.
  • El backend valida JWT y persiste en PostgreSQL (SesionChat, y opcional ChatMessage).
  • Aplica rate limiting por IP/usuario en backend para proteger a Rasa/Actions.

8) ¿Qué tocar al integrar con el backend?​

  • URL interna de Rasa para el backend (por ejemplo, http://rasa-core:5005 si comparten red).
  • Endpoint REST:
    POST /webhooks/rest/webhook, body: 
    { "sender": "<sessionId|userId>", "message": "<texto del usuario>" }

9) Operación / Mantenimiento​

⚠️ Asegúrate de estar en la carpeta del proyecto ~/projects/chatbot/ antes de ejecutar los comandos.

Desplegar / Reiniciar todo​

sudo docker-compose down
sudo docker-compose up -d --build

10) Errores frecuentes y cómo detectarlos​

  • Rasa no llama a acciones → Revisa endpoints.yml (URL de actions) y domain.yml (acciones declaradas).

  • Fallback siempre → Umbrales (FallbackClassifier.threshold / RulePolicy.core_fallback_threshold) demasiado altos, dataset pobre o desbalanceado.

  • LLM tarda en la primera respuesta → Warm-up de ONNX; considera precalentar en __init__ o al arrancar el servicio.

  • Respuestas fuera de tema → Reforzar prompt en actions.py, bajar temperature, revisar límites del fallback.

  • Modelos no cambian al reiniciar → Confirmar que el nuevo .tar.gz está en models/ y que Rasa se reinició.
