Módulos Principales

Esta sección describe qué hace cada módulo, qué servicios intervienen y qué debe conocer un desarrollador para mantener o extender la app. No repite implementación línea a línea: se enfoca en responsabilidades, flujos y puntos de integración.

Autenticación y gestión de sesión​

Propósito: validar credenciales, emitir/renovar token y mantener la sesión segura del usuario.

Servicios clave: auth_service.dart, connectivity_service.dart (verificación de red), notification_service.dart (registro FCM tras login), firebase_messaging_handler.dart (recepción de mensajes).

Responsabilidades principales

  • Inicio de sesión, cierre de sesión.
  • Almacenamiento seguro de token, publicId, deviceId en SecureStorage.
  • Renovación/validación de sesión (token) de forma transparente.
  • Registro/actualización del token FCM tras un login exitoso.

Entradas / Salidas

  • Entrada: usuario, contraseña, appVersion.
  • Salida: objeto de sesión (ids + token) y estado autenticado en la UI.

Errores comunes

  • 401/403: limpiar sesión y redirigir a Login.
  • Timeout/red: informar al usuario y permitir reintentos.
  • Incompatibilidad de versión: bloquear con mensaje claro si el backend lo exige.

Seguridad

  • Token en cabecera para todas las llamadas autenticadas.
  • Nunca registrar en logs token ni contraseñas.
  • Usar SecureStorage para persistir credenciales.

Gestión de pacientes​

Propósito: CRUD de pacientes y acceso a su información clínica relevante para la app.

Servicios clave: paciente_service.dart, archivo_service.dart (adjuntos), resultado_service.dart (resultados vinculados).

Responsabilidades principales

  • Crear/actualizar datos del paciente asociados a la cuenta de usuario.
  • Consultar información resumida (dashboard) y detalle bajo demanda.
  • Manejo de adjuntos clínicos básicos (según permisos).

Entradas / Salidas

  • Entrada: filtros de búsqueda, identificadores del paciente.
  • Salida: modelos de paciente y colecciones para renderizar en UI.

Offline

  • Cachear consultas recientes y permitir lectura limitada sin conexión.

Chat y sesiones de chat​

Propósito: brindar un canal de comunicación (con personal de salud o bot) asociado a la cuenta/episodio del paciente.

Servicios clave: chat_service.dart, sesion_chat_service.dart, notification_service.dart (notificaciones nuevas/replies).

Responsabilidades principales

  • Crear/gestionar sesiones de chat y su historial.
  • Enviar/recibir mensajes, manejar estados (entregado/leído).

Consideraciones

  • El chat debe reaccionar a push (mensaje nuevo) mediante firebase_messaging_handler.dart → notification_state.dart → refresco de UI.
  • Manejo de adjuntos: usar archivo_service.dart si aplica.

Notificaciones push (FCM)​

Propósito: informar eventos críticos (resultados disponibles, recordatorios, nuevos recursos o mensajes).

Servicios clave: notification_service.dart, firebase_messaging_handler.dart, notification_state.dart.

Flujo resumido

  1. Backend envía notificación → FCM.
  2. FCM entrega al dispositivo (foreground/background/terminated).
  3. firebase_messaging_handler.dart normaliza el mensaje.
  4. notification_state.dart actualiza la UI (banners, badges, navegación).

Buenas prácticas

  • Registrar/actualizar el token FCM en login.
  • Manejar deep-links o rutas para abrir pantallas específicas.
  • Registrar interacciones del usuario (tocar notificación, descartar).

Encuestas y formularios​

Propósito: recolectar datos de salud y socioeconómicos con formularios estructurados.

Servicios clave: encuesta_service.dart, inf_socioeconomica_service.dart.

Responsabilidades principales

  • Renderizar formularios y validar campos.
  • Guardar borradores en cache para modo offline.
  • Enviar respuestas al backend y manejar estados (pendiente/enviado).

Errores y recuperación

  • Sin conexión: guardar en cola para sincronización posterior.
  • Validación: feedback rápido en campo (requeridos, formatos).

Resultados y recursos médicos​

Propósito: exponer resultados de exámenes y materiales de apoyo (recursos, guías).

Servicios clave: resultado_service.dart, resource_service.dart.

Responsabilidades principales

  • Consultar listado y detalle de resultados (con control de acceso).
  • Descargar/visualizar recursos complementarios (PDFs, guías).

UI/UX sugerido

  • Distinción visual por estado del resultado (nuevo, visto).
  • Acciones rápidas: ver, descargar, compartir (si la política lo permite).

Ubicación y mapas​

Propósito: soporte de geolocalización para flujos específicos (p.ej., localización de centros, trazabilidad de dispositivos o eventos).

Servicios clave: ubicacion_service.dart.

Responsabilidades principales

  • Obtener coordenadas del dispositivo (previo consentimiento).
  • Enviar/consultar ubicaciones asociadas a entidades (usuario, evento, recurso).

Privacidad

  • Solicitar permisos explícitos y justificar su uso en la UI.
  • Minimizar persistencia local de geodatos y aplicar políticas de retención.

Archivos y adjuntos​

Propósito: subir y descargar archivos clínicos vinculados a formularios y resultados.

Servicios clave: archivo_service.dart, resource_service.dart (recursos estáticos), connectivity_service.dart (reintentos).

Flujos típicos

  • Upload desde formularios o chat (si aplica), con barra de progreso y reintentos.
  • Download de informes/guías con manejo de permisos del SO.

Recomendaciones

  • Limitar tamaños y tipos de archivos permitidos por política.
  • Reintentos exponenciales en fallos de red.
  • Limpiar temporales tras completar operaciones.

Checklist para desarrolladores

  • ¿Este flujo requiere autenticación? → usar auth_service + token en cabeceras.
  • ¿Debe funcionar offline? → cache/borradores + cola de sincronización.
  • ¿Se deben disparar notificaciones o deep-links? → notification_service + handler.
  • ¿Hay datos sensibles? → cifrado en tránsito, SecureStorage en reposo.
  • ¿Se necesitan permisos del SO? → solicitar, justificar, degradar funcionalidad si se niegan.
